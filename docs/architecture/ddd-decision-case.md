# Octo 架构决策案例

> **项目**：Octo - 服务知识，触手可及
> **案例类型**：DDD 架构设计
> **日期**：2026-03-14
> **参与者**：架构师、Claude AI Assistant

---

## 1. 项目背景

### 1.1 初始需求

Octo 是一个架构师的个人提效工具，辅助架构师快速高质量完成需求方案分析和设计。核心功能包括：

1. **服务知识管理**：自动分析 Spring Boot 微服务，提取职责、接口、依赖等知识
2. **设计空间管理**：架构师的个人工作空间，支持历史设计管理、参考资料管理、模板沉淀
3. **AI 协作支持**：通过 MCP 协议与 Claude Code 集成，实现人机协作设计

### 1.2 技术约束

- **后端**：Spring Boot 3.x, Java 17+
- **数据库**：PostgreSQL 15+
- **前端**：React 18+, Ant Design 5.x
- **MCP**：JSON-RPC 2.0

## 2. 面临的问题

### 2.1 初始困惑

在讨论架构时，我们发现了一些关键问题：

1. **如何组织代码？**
   - 问题：没有清晰的模块边界
   - 后果：代码容易耦合，职责不清

2. **业务逻辑放在哪里？**
   - 问题：Controller、Service、Repository 都可以放业务逻辑
   - 后果：代码复用性差，测试困难

3. **数据访问用什么？**
   - 问题：JPA 和 MyBatis-Plus 都在考虑中
   - 后果：需要权衡优缺点

4. **是否需要 DDD？**
   - 问题：DDD 概念复杂，不知道如何应用到这个项目
   - 后果：过度设计风险 vs 清晰架构收益

5. **如何保持代码质量？**
   - 问题：缺少统一的编码和测试规范
   - 后果：代码风格不统一，测试覆盖难保证

## 3. 探索的方案

### 3.1 代码组织方案

#### 方案 A：按功能模块划分（推荐）
```
backend/src/main/java/com/skm/
├── service-knowledge/
│   ├── domain/
│   ├── application/
│   ├── infrastructure/
├── design-space/
│   ├── skill-management/
│   └── mcp-integration/
```

**优点**：
- 清晰的业务边界
- 团队容易理解
- 便于并行开发

**缺点**：
- 需要明确每个模块的职责

#### 方案 B：按技术层划分
```
backend/src/main/java/com/skm/
├── controller/
├── service/
├── repository/
└── entity/
```

**优点**：
- 简单直接
- 新人容易上手

**缺点**：
- 业务边界不清晰
- 容易产生跨层依赖

**决策**：选择方案 A（按限界上下文）

---

### 3.2 业务逻辑分层方案

#### 方案 A：完整 DDD（领域方法、聚合、值对象）
```
- Entity（实体）
- Value Object（值对象）
- Aggregate（聚合）
- Domain Service（领域服务）
- Repository（仓储接口）
- Domain Event（领域事件）
```

**优点**：
- 模型表达能力最强
- 业务规则内聚
- 适合复杂业务

**缺点**：
- 学习成本高
- 开发效率较低
- 可能过度设计

#### 方案 B：轻量级 DDD（推荐）
```
- Entity（实体包含基本验证）
- Repository（仓储接口）
- Application Service（编排用例）
- DTO（数据传输对象）
```

**优点**：
- 开发效率高
- 易于理解和维护
- 适合当前项目规模
- 清晰的分层

**缺点**：
- 领域建模能力较弱
- 某些复杂业务需要额外处理

**决策**：选择方案 B（轻量级 DDD）

**理由**：
1. Octo 是 MVP 阶段，业务相对简单
2. 团队规模小，不需要复杂的领域建模
3. 快速迭代更重要
4. 后续可以升级到完整 DDD

---

### 3.3 ORM 框架选择方案

#### 方案 A：MyBatis-Plus
**优点**：
- SQL 控制力强
- 复杂查询支持好
- 国内生态成熟
- 适合复杂 SQL

**缺点**：
- 需要手写 SQL
- 维护成本较高
- 学习曲线陡峭

#### 方案 B：Spring Data JPA（推荐）
**优点**：
- 开发效率高（自动 CRUD）
- 学习曲线平缓
- 与 Spring Boot 无缝集成
- 事务管理简单
- 社区成熟
- 基本 CRUD 性能足够

**缺点**：
- 复杂查询能力较弱
- SQL 优化需要额外配置

**决策**：选择 Spring Data JPA

**理由**：
1. Octo 的数据访问主要是标准 CRUD
2. PostgreSQL JSONB 支持方便
3. 团队更熟悉 JPA
4. MVP 快速开发更重要
5. 复杂查询可以使用 @Query 注解或原生 SQL（按需）

## 4. 最终决策

### 4.1 限界上下文划分

**4 个限界上下文**：

| 上下文 | 核心领域 | 职责 | 外部依赖 |
|--------|---------|------|----------|
| Service Knowledge | 服务知识市场管理 | CRUD、发布、检索、搜索 | PostgreSQL、Design Space、MCP |
| Design Space | 架构师个人工作空间 | 创建、订阅、文件系统、Git 集成 | PostgreSQL、文件系统、Git |
| Skill Management | 技能配置和管理 | 安装、卸载、配置、多平台适配 | PostgreSQL |
| MCP Integration | MCP 协议实现 | 工具注册、请求分发、响应 | 4 个上下文 |

**关键设计原则**：
1. 每个上下文有清晰的业务边界
2. 上下文之间通过应用服务同步通信
3. MCP Integration 作为技术上下文，连接其他业务上下文

### 4.2 DDD 轻量级原则

**采用**：轻量级 DDD

**关键原则**：
1. ✅ 清晰的限界上下文
2. ✅ 高内聚低耦合
3. ✅ Service 层业务逻辑封装
4. ✅ Repository 只负责数据访问
5. ❌ 不使用值对象（用基本类型）
6. ❌ 不使用领域方法（业务逻辑在 Service）
7. ❌ 不使用领域规范（验证在 Service）
8. ❌ 不使用聚合根（事务管理在 Service）

**代码结构**：
```
{context}/
├── domain/              # 实体（包含基本验证）
├── repository/         # 仓储接口
├── application/         # 应用服务、控制器
├── infrastructure/     # 基础设施实现
```

### 4.3 ORM 技术栈

**选择**：Spring Data JPA

**理由**：
1. MVP 阶段适合快速开发
2. 标准 CRUD 操作效率高
3. PostgreSQL JSONB 支持好
4. 团队熟悉度高
5. 事务管理简单
6. 社区成熟

**PostgreSQL JSONB 支持**：
```java
@Type(type = "jsonb")
@Column(columnDefinition = "jsonb")
private String knowledge;
```

## 5. 实施建议

### 5.1 开发优先级

**Phase 1（MVP）**：
1. Service Knowledge Context - 基础 CRUD
2. Design Space Context - 基础 CRUD + 文件系统
3. MCP Integration Context - 基础工具暴露

**Phase 2（后续）**：
4. Skill Management Context - 技能管理
5. 高级功能 - 复杂查询、性能优化

### 5.2 风险和缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 轻量级 DDD 过于简化 | 后期重构成本 | 持续评估，必要时升级到完整 DDD |
| JPA 复杂查询能力弱 | 某些功能实现困难 | 使用 @Query 或原生 SQL 作为补充 |
| 限界上下文划分不够合理 | 跨上下文耦合 | 持续优化上下文边界 |
| 缺少完整 DDD 经验 | 团队学习成本 | 文档充分，代码注释清晰 |

## 6. 关键经验总结

### 6.1 架构设计经验

#### ✅ 成功经验

1. **清晰的业务边界至关重要**
   - 按限界上下文划分让代码职责清晰
   - 每个上下文可以独立开发和测试

2. **适度胜过完美**
   - 轻量级 DDD 适合当前项目规模
   - 避免过度设计影响交付速度

3. **渐进式演进**
   - 先用简单方案快速实现
   - 为后续升级预留空间
   - MVP 优先于完整性

4. **讨论驱动决策**
   - 充分讨论不同方案
   - 记录决策理由
   - 确保团队理解一致

#### ⚠️ 改进空间

1. **提前考虑限界上下文**
   - 初始只考虑了 3 个，后来补充到 4 个
   - 建议：在架构设计阶段绘制完整的影响图

2. **技术选型需要更多实践数据**
   - JPA vs MyBatis 的选择基于理论分析
   - 建议：通过 POC 验证性能和开发效率

3. **文档编写同步进行**
   - 架构设计、编码规范、测试规范分批完成
   - 更容易保持一致性和完整性

### 6.2 沟通协作经验

#### ✅ 成功经验

1. **结构化讨论流程**
   - 从问题识别开始
   - 逐个分析方案
   - 明确记录决策和理由
   - 最终方案总结

2. **提问技巧**
   - 先提出自己的想法，询问用户意见
   - 提供 2-3 个选项，而不是开放性问题
   - 每个选项说明优缺点

3. **及时确认**
   - 在每个决策点明确确认
   - 使用工具（如 AskUserQuestion）确保用户选择

#### ⚠️ 改进空间

1. **可以更早引入可视化**
   - 使用图表展示限界上下文
   - 可视化帮助快速理解复杂关系

2. **可以记录更多中间思考**
   - 记录探索过程中的试错
   - 展示权衡的思考路径

## 7. 可复用的决策框架

### 7.1 限界上下文划分框架

1. **识别核心业务领域**
   - 问题：主要业务实体有哪些？
   - 示例：服务知识、设计空间、技能、MCP

2. **定义上下文边界**
   - 问题：哪些实体属于同一个业务领域？
   - 判断依据：一起变更、一起使用、强业务关联

3. **明确上下文间关系**
   - 问题：上下文之间如何通信？
   - 选项：同步调用、异步事件

4. **确定外部依赖**
   - 问题：每个上下文依赖哪些外部系统？
   - 分类：数据库、文件系统、第三方 API

### 7.2 技术选型框架

1. **明确项目约束**
   - 团队技能栈、学习成本、交付时间
   - 技术约束：性能、兼容性、生态要求

2. **列出候选方案**
   - 识别 2-3 个可行方案
   - 每个方案列出关键优缺点

3. **评估维度**
   - 开发效率（学习成本、开发速度）
   - 技术能力（能否满足需求）
   - 维护性（社区支持、文档质量）
   - 性能（吞吐量、延迟、资源占用）
   - 生态成熟度（社区活跃度、第三方库）

4. **权重和评分**
   - 为每个维度分配权重
   - 逐个方案评分
   - 计算总分

5. **决策和记录**
   - 选择最优方案
   - 记录详细决策理由
   - 定期回顾评估
